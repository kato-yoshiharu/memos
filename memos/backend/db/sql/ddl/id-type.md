# Primary Key / ID の型選択

データベースの主キー・IDカラムに使用する型の比較検討。

## 選択肢

- SERIAL
- UUID
- ULID

## UUID (Universally Unique Identifier)

### メリット

- 標準化されており（RFC 4122）、どのDBでもネイティブサポート
- 衝突確率が極めて低い
  - 分散システムでも独立して生成可能

### デメリット

- サイズが大きい（128 bit）
- v4：ランダム生成で非順序のためインデックス効率が悪い
- v1：マシン時刻に依存
- v3/v5：生成に名前空間が必要で自由度が低い

## ULID (Universally Unique Lexicographically Sortable Identifier)

### メリット

- UUIDと同じサイズだが、ソート可能（左側がタイムスタンプで右側がランダム）
  - インデックス効率が良い（単調増加）
  - 作成順序が保証される

### デメリット

- 標準化がまだ進行中（RFC 9562では指定なし）のため、DBのネイティブサポートが限定的
- タイムスタンプ依存（マシン時刻の影響を受ける）

## SERIAL / BIGSERIAL

### メリット

- サイズが小さい（32 bit / 64 bit、最小規模）
- 生成が高速
- DBのネイティブサポート

### デメリット

- 予測可能（セキュリティリスク）
- 複数DB間の同期が複雑
- 単一ボトルネック
