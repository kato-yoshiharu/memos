# FSD (Feature-Sliced Design)

## 概要

Feature-Sliced Design (FSD) は、中大規模フロントエンドのためのアーキテクチャパターン。
Feature(機能)単位でスライスに分け、その中にセグメントを切っていく。

## 主な特徴

### 1. レイヤーベースの構造

プロジェクトを複数のレイヤーに分割し、各レイヤーが明確な責務を持つ：

- app
  - アプリケーション全体の初期化、グローバル設定
- pages
  - ページコンポーネント
- widgets
  - features や entities を組み合わせた、自己完結したUI単位で、完全なユースケースを実装する大型コンポーネント
- features
  - 機能を実現するためのモジュール
- entities
  - ビジネスエンティティ
- shared
  - プロジェクト固有の知識に依存しない、プロジェクト横断で再利用可能なユーティリティ、コンポーネント、定数

### 2. スライスとセグメント

各機能は「スライス」として独立した単位で管理：

```
features/
├── auth/
│   ├── api/        - API呼び出し
│   ├── model/      - ビジネスロジック、状態管理
│   ├── ui/         - UIコンポーネント
│   └── index.ts    - 公開インターフェース
├── comments/
│   ├── api/
│   ├── model/
│   ├── ui/
│   └── index.ts
```

### 3. セグメントの役割

- api
  - データフェッチロジック
- model
  - ビジネスロジック、状態管理
- ui
  - UIコンポーネント
- lib
  - このスライス固有のユーティリティ
- types
  - 型定義
- index.ts
  - 公開API

## メリット・デメリット

### メリット

- 高凝集度 - 関連コードが集約。保守とテストが容易
- 低結合度 - 機能が独立。スケーラビリティと並列開発が可能

### デメリット・注意点

- 初期学習コストが高い
- 小規模プロジェクトでは過設計になる可能性
- 依存関係の循環を防ぐための厳格な規則が必要
- 規則を破ると全体が崩れやすい

## 依存関係のルール

FSDの核となる規則：**レイヤー間・スライス間の依存方向を厳密に管理**

1. レイヤー間の依存方向: app → pages → widgets → features → entities → shared
2. スライス同士は直接依存しない: shared経由
3. セグメント間の依存方向: api → model → ui

- 下位レイヤーは上位レイヤーに依存しない
- スライス同士は直接依存しない（shared経由）
- index.tsで明確に公開APIを定義

## ツール・ライブラリサポート

- **ESLint plugin** - 依存関係の違反を検出
